# AI 工程师面试 1 个月冲刺计划

## 1. 目标

你的目标不是在 1 个月内“学完 AI”，而是：

**在 1 个月内做出一个可演示、可讲透、可写进简历的 RAG 项目，并据此支撑 AI 工程师面试。**

面试官最终会判断三件事：

1. 你有没有真实工程能力
2. 你有没有做过完整的 AI 应用项目
3. 你能不能把后端经验迁移到 AI 场景

所以这 1 个月的任务不是追求技术栈最全，而是产出最强的面试资产。

---

## 2. 技术路线

考虑到你的背景和时间限制，建议技术栈固定为：

1. 主语言：`Java`
2. 服务框架：`Spring Boot`
3. 主存储：`PostgreSQL`
4. 向量能力：`pgvector`
5. 缓存：`Redis`
6. 模型调用：OpenAI 兼容 API
7. Python 角色：只用于评测脚本、实验脚本、数据处理脚本

当前阶段不建议优先引入：

1. `Elasticsearch`
2. `Milvus`
3. 复杂 Agent 框架
4. 本地大模型部署
5. 过重的多组件架构

原因很简单：

**你现在的瓶颈不是组件不够多，而是要在 1 个月内把项目做完整、讲完整。**

---

## 3. 项目定位

项目建议定位为：

**企业知识库 RAG 问答系统**

项目必须覆盖的能力：

1. 文档上传
2. 文档解析
3. 文档切块
4. 向量化入库
5. 检索召回
6. 大模型回答
7. 引用来源展示
8. 文档元数据管理
9. 问答记录保存
10. 基础评测

项目不要一开始做得过重。  
你当前最重要的是做出一个完整闭环：

**上传文档 -> 建索引 -> 提问 -> 检索 -> 回答 -> 展示引用**

---

## 4. 为什么当前优先 PostgreSQL

你这次冲刺阶段，优先 `PostgreSQL` 而不是 `Elasticsearch`，不是因为 ES 不好，而是因为：

1. `PostgreSQL` 可以统一承载主数据
   - 知识库
   - 文档元数据
   - Chunk
   - 问答记录
   - 任务状态
2. 加上 `pgvector` 就能完成基础语义检索
3. 系统组件更少，开发和排障成本更低
4. 面试时更容易讲清楚系统边界和设计取舍

你可以把 `Elasticsearch` 作为后续优化方向，但不是现在的必选项。

面试时可以这样回答：

**第一阶段我优先选择 PostgreSQL，是为了降低系统复杂度，先把主数据管理、向量检索和问答链路统一起来，快速完成端到端闭环；如果后续数据规模和全文检索要求上来，再引入 Elasticsearch 做 BM25 和混合检索。**

---

## 5. 1 个月交付物

在面试前，你至少要完成这些东西：

1. 一个能跑起来的项目仓库
2. 一份清晰的 README
3. 一张系统架构图
4. 一套接口说明
5. 一份简历项目描述
6. 一份项目难点与优化点总结
7. 一套基础问答测试集
8. 一份 10 到 15 分钟项目讲稿

如果这些没有完成，就不算准备结束。

---

## 6. 四周计划

## 第 1 周：搭骨架，跑通数据链路

目标：把项目基础架构搭起来，先把数据流做通。

本周必须完成：

1. 定项目目录结构
   - `src/main/java/com/example/rag/controller`
   - `src/main/java/com/example/rag/service`
   - `src/main/java/com/example/rag/ingestion`
   - `src/main/java/com/example/rag/retrieval`
   - `src/main/java/com/example/rag/generation`
   - `src/main/java/com/example/rag/integration`
   - `src/main/java/com/example/rag/repository`
   - `src/main/java/com/example/rag/model`
   - `src/main/java/com/example/rag/config`
   - `src/main/java/com/example/rag/common`
   - `src/main/resources/db/migration`
   - `work`
2. 设计数据库表
   - knowledge_base
   - document
   - document_chunk
   - chat_session
   - chat_message
   - indexing_task
3. 选定文档样本
   - Markdown
   - PDF
   - txt
4. 完成文档上传与存储
5. 完成文档解析与切块
6. 完成 chunk 入库
7. 画出第一版架构图

本周验收标准：

1. 可以上传文档
2. 可以把文档切成 chunk
3. chunk 和元数据能写入 PostgreSQL
4. 你能讲清楚数据库设计和数据流

## 第 2 周：跑通问答主链路

目标：完成从提问到回答的端到端流程。

本周必须完成：

1. 接入 embedding 模型
2. 用 `pgvector` 保存向量
3. 实现 TopK 检索
4. 设计 Prompt 模板
5. 接入大模型回答
6. 返回引用来源
7. 保存问答记录
8. 提供基础接口
   - 上传文档
   - 建索引
   - 提问问答
   - 查看问答历史

本周验收标准：

1. 可以针对知识库提问
2. 可以返回引用片段和最终答案
3. 问答记录可查
4. 你能把整条 RAG 链路讲出来

## 第 3 周：补工程化与基础优化

目标：让项目从“能跑”变成“像样”。

本周必须完成：

1. 做异步建索引
2. 增加任务状态追踪
3. 增加失败重试机制
4. 增加基础缓存
5. 增加结构化日志
6. 增加基础配置管理
7. 尝试 2 到 3 组切块参数对比
8. 做 10 到 20 条测试问答

本周验收标准：

1. 你能说出至少 2 个工程难点
2. 你能说出至少 2 个优化点
3. 你能展示优化前后的效果差异

## 第 4 周：面试化包装

目标：把项目变成真正的面试资产。

本周必须完成：

1. 完善 README
2. 完善架构图
3. 补充接口文档
4. 整理项目难点
5. 整理优化点
6. 写简历项目描述
7. 写面试问答稿
8. 做 3 次项目自述模拟

本周验收标准：

1. 能做 10 分钟项目介绍
2. 能回答 20 个项目深挖问题
3. 简历上每一句描述都有项目证据支撑

## 7. 本项目必须讲透的问题

下面这些问题，面试时你必须能脱稿回答：

1. 你为什么做这个项目
2. 这个项目解决什么问题
3. 系统整体架构是什么
4. 为什么主存储选 PostgreSQL
5. 为什么当前阶段不先引入 Elasticsearch
6. 文档怎么解析和切块
7. Chunk 为什么这样设计
8. 向量检索怎么做
9. 如何降低幻觉
10. 如何返回引用来源
11. 如何保存问答记录和任务状态
12. 如何处理文档更新
13. 如何优化响应时间
14. 如何优化成本
15. 如果后续规模扩大，怎么扩展到 Elasticsearch 或混合检索
16. 这个项目最难的部分是什么
17. 这个项目最有价值的工程设计是什么
18. 如果重做一次，你会怎么改

你的回答必须覆盖五个维度：

1. 业务目标
2. 技术方案
3. 工程取舍
4. 数据结果
5. 风险边界

## 8. 简历策略

你不应该把自己包装成“纯算法工程师”，而应该定位成：

**有强后端工程背景，转向 LLM / RAG 应用落地的 AI 工程师**

简历里要强调的不是“我会调模型接口”，而是：

1. 我能设计完整的数据链路
2. 我能做稳定的服务系统
3. 我能做文档处理、检索、生成和评测闭环
4. 我能做工程取舍，而不是堆框架

项目描述建议结构：

1. 项目背景
2. 你的职责
3. 核心设计
4. 技术难点
5. 量化结果

---

## 9. 本月执行纪律

这个月你要严格遵守下面的顺序：

1. 先做主链路
2. 再做工程化
3. 再做优化
4. 最后做简历和模拟面试

本月不要陷入这些坑：

1. 一开始就追求复杂架构
2. 一开始就研究过多框架
3. 项目没跑通就开始包装简历
4. 没有数据支撑就空谈优化
5. 花太多时间在次要技术选型上

---

## 11. 最终目标

一个月后，你应该达到的状态是：

1. 你有一个完整可演示的 RAG 项目
2. 你能讲清楚每个关键设计决策
3. 你能把 Java 后端经验自然迁移到 AI 系统场景
4. 你有足够材料支撑 AI 工程师面试

这份 `plan.md` 是执行文档，不是阅读材料。  
之后每周都按验收标准检查，不要只看进度，要看是否真的形成了面试资产。
