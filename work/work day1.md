# Day 1：定边界

## 项目名

云服务结算知识库问答系统

## 一句话描述

面向产品、开发、测试与运维团队，构建一个基于 RAG 的企业知识库问答系统，将结算领域的设计文档、运维手册和流程文档沉淀为可检索、可引用、可追溯的智能问答能力。

## 项目定位

这是一个面向内部团队使用的企业知识库 RAG 系统，第一阶段聚焦于跑通文档入库和知识检索主链路，不追求复杂 Agent 能力，也不引入过重的多组件架构。

核心目标不是“把模型能力做炫”，而是做出一个可演示、可讲透、可写进简历的完整 AI 应用项目。

## 目标用户

1. 产品经理
2. 开发工程师
3. 测试工程师
4. 运维工程师

## 核心问题

团队内结算相关知识分散在概要设计、详细设计、接口文档、运维手册和问题记录中，存在以下问题：

1. 文档分散，信息检索成本高
2. 历史方案和规则依赖资深同事口口相传
3. 遇到线上问题时，定位相关文档和关键段落效率低
4. 新成员熟悉业务周期长

因此需要一套能够基于企业内部文档进行检索增强生成的知识问答系统，在回答问题时给出引用来源，提升知识获取效率和答案可信度。

## 第一阶段 MVP 范围

## 面试可用 Q&A

### 1. 这个项目给谁用

这个项目面向内部知识密集型团队使用，主要用户包括产品、开发、测试和运维。它解决的不是开放域问答，而是企业内部结算领域知识的沉淀、检索和复用问题。

### 2. 解决什么问题

它主要解决三类问题：第一，内部文档分散，检索成本高；第二，很多关键知识依赖资深同事经验，难以沉淀；第三，排查问题或理解业务规则时，无法快速定位相关文档和依据。这个系统希望把“找资料”和“理解资料”的过程合并，提升团队知识获取效率。

### 3. 为什么是 RAG

因为这个场景的核心不是让模型凭空生成，而是让模型基于企业内部文档回答问题。RAG 的价值在于先检索再生成，把答案建立在已有知识之上，同时返回引用片段，降低幻觉风险，也更适合企业场景对可解释性和可追溯性的要求。

### 4. 为什么用 Java

主服务选择 Java，主要是因为我原本就是 Java 后端出身，使用 Spring Boot 可以更快搭建稳定的工程骨架，把精力放在文档处理、检索链路和系统设计上，而不是重新学习一整套后端基础设施。对我当前的求职目标来说，重点是证明我能把后端工程能力迁移到 AI 应用场景。

### 5. 为什么当前只用 PostgreSQL + pgvector

第一阶段优先使用 PostgreSQL + pgvector，是为了降低系统复杂度。PostgreSQL 可以统一承载知识库、文档、chunk、问答记录和任务状态，加上 pgvector 就能完成基础语义检索。这样可以先快速跑通端到端链路，等后续数据规模和全文检索需求上来，再考虑引入 Elasticsearch 做 BM25 或混合检索。

### 6. 为什么第一阶段不先做复杂问答能力

因为当前最重要的不是堆功能，而是先做出完整闭环。如果上传、解析、切块、入库这些基础能力不稳定，后面的问答效果就没有可靠基础。第一阶段先把数据链路做扎实，后续再逐步补 embedding、召回、生成和评测。

## 当前边界结论

第一周的重点不是“模型回答得多聪明”，而是把文档数据链路跑通，并把系统模块、数据模型和设计取舍讲清楚。

当前项目的第一阶段闭环应该是：

文档上传 -> 文档解析 -> 文本切块 -> chunk 入库 -> 向量化预留 -> 检索链路预留

只有这个闭环稳定了，第二周再接 embedding、向量检索和大模型回答，项目推进才会顺。
