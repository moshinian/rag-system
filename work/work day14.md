# Day 14：端到端联调验收

## 当前结论

Day 14 的目标不是再补新的主功能，而是：

**把 Week 2 现有的问答主链路做一次完整验收，并把真实结果固定下来。**

经过 Day 13，项目已经具备：

1. 知识库创建
2. 文档上传
3. 文档解析
4. 文本切块
5. chunk 入库
6. chunk embedding
7. pgvector 检索
8. Prompt 组装
9. OpenAI-compatible chat completion
10. `/qa/ask`
11. `sources` 结构化来源
12. 问答记录持久化
13. `/qa/history`

所以 Day 14 的核心不是“继续堆功能”，而是：

**验证主链路稳定性、可解释性和演示完整度。**

## Day 14 要完成什么

今天建议完成下面几件事：

1. 做一次从上传到问答历史的完整联调
2. 固定一组可复现的验收样例
3. 检查返回结构是否稳定
4. 检查失败路径和空结果路径
5. 更新 README 和进度文档

## Day 14 验收范围

建议按下面顺序验收：

1. 创建知识库
2. 上传文档
3. 触发解析与分块
4. 触发 embedding
5. 调用 `/qa/retrieve`
6. 调用 `/qa/ask`
7. 调用 `/qa/history`

每一步至少确认：

1. 接口返回成功
2. 关键字段完整
3. 数据状态正确
4. 能衔接到下一步

## Day 14 重点检查项

今天要重点看这几类问题：

1. 检索结果是否稳定
2. 回答是否严格基于召回内容
3. `sources` 和 `retrievalResults` 是否一致
4. 问答记录是否写库成功
5. 历史查询是否能查回最新记录
6. 空召回时的回答是否符合预期

## Day 14 输出物

今天结束后，最好至少产出：

1. 一组完整 curl 样例
2. 一次真实联调结果说明
3. README 中的验收章节
4. 当前项目状态更新

## Day 14 完成后的项目状态

如果 Day 14 顺利完成，Week 2 的最小 RAG 闭环就算完成第一版收口：

```text
知识库创建 -> 文档上传 -> 解析 -> 切块 -> chunk 入库 -> chunk 向量写库 -> query embedding -> TopK 检索 -> Prompt 组装 -> LLM 回答 -> 来源返回 -> 问答记录持久化 -> 历史查询 -> 端到端验收
```

下一步就可以考虑：

1. 来源引用格式化
2. session 复用
3. 多轮对话
4. 问答效果优化
