# Day 21：Week 3 验收与文档收口

## 今日目标

Day 21 不再扩新功能，只做一件事：

**把 Week 3 已经完成的异步索引、重试恢复、结构化日志、配置整理和中文评测统一收口。**

这一天的重点不是再补能力，而是把已经做完的内容整理成可以交付、可以复盘、可以继续进入 Week 4 的状态。

## 为什么 Day 21 要收口

到 Day 20 为止，Week 3 已经完成：

1. 异步索引
2. 任务状态追踪
3. 手动重试与卡住任务恢复
4. 结构化日志
5. 配置梳理与参数外置
6. 切块参数实验
7. 中文问答评测集与第一版真实评测

如果 Day 21 继续扩功能，会带来两个问题：

1. README、状态文档、评测结果和真实代码容易再次脱节
2. Week 3 的成果不容易形成清晰的阶段结论

所以 Day 21 只做验收和文档统一，不再新增大功能。

## Day 21 要完成什么

### 1. README 收口

需要确认：

1. README 只描述当前已经实现的能力
2. Week 3 的完成项和边界说清楚
3. 中文评测结果被纳入当前状态说明

### 2. 状态文档收口

需要确认：

1. `current-status.md`
2. `week3.md`
3. `work day20.md`

这几份文档之间的结论一致，不互相打架。

### 3. Week 3 验收样例整理

需要至少保留：

1. 一组异步索引成功样例
2. 一组失败重试或恢复说明
3. 一组中文问答评测样例

### 4. 风险与后续方向

需要明确指出：

1. 无答案场景下的召回抑制仍需优化
2. 批量编排和任务取消还未开始
3. 更完整的监控、指标和 tracing 还未开始

## Day 21 的验收口径

Day 21 结束时，至少要能回答清楚下面这些问题：

1. Week 3 到底完成了什么
2. 哪些能力是真实联调过的
3. 哪些地方还只是夹具、模板或后续方向
4. 当前系统能否作为最小可用 RAG 服务继续往下推进

## 输出物

Day 21 完成后，至少应该留下：

1. 收口后的 README
2. 收口后的 `current-status.md`
3. 收口后的 `week3.md`
4. 一版 Week 3 验收结论

## 边界说明

Day 21 不做：

1. 新接口
2. 新数据库迁移
3. 新异步编排能力
4. 新模型接入

如果有新功能想法，统一放到 Week 4 再处理。
